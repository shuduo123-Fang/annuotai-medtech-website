# Project 工作区规则

## 1. 项目边界

本文件所在目录是当前项目的根目录。所有与本项目相关的代码、文档、资料和输出物，都应放在本项目文件夹内。

不得将本项目文件直接散放到上级 Workspace 根目录，也不得放到电脑其他无关位置。

## 2. 上级全局规范

本项目遵循上级 Workspace 的全局工作规范。开始新的项目型、复杂、长期或交付物任务前，应先参考：

- `../全局工作台.md`
- `../全局复利与踩坑日志.md`
- `../新项目创建 SOP.md`

简单问答、小修小改、临时查询不强制执行上述检查。

## 3. 基础目录

项目目录建议保持如下结构：

```text
Project-###-项目名称/
├── AGENTS.md
├── README.md
├── docs/
├── src/
├── assets/
└── outputs/
```

用途说明：

- `README.md`：项目目标、范围、状态、关键决策。
- `docs/`：需求、方案、笔记、过程文档。
- `src/`：代码、脚本、配置等可执行或可复用内容。
- `assets/`：图片、参考资料、素材、数据样本。
- `outputs/`：最终交付物、导出文件、报告、成品。

## 4. 工作规则

- 开始工作前，先确认任务是否属于本项目范围。
- 新增文件时，优先放入对应目录，不在项目根目录堆放过程文件。
- 交付给用户的最终文件放入 `outputs/`。
- 过程文档、分析记录、需求说明放入 `docs/`。
- 代码、脚本、配置放入 `src/`。
- 素材、图片、数据样本、参考资料放入 `assets/`。
- 不删除或覆盖已有内容，除非用户明确要求。
- 对已有文件做修改时，保持改动最小化，避免无关重构和无关清理。

## 5. 项目记录

当项目出现重要决策、踩坑、可复用方法或后续注意事项时，应记录到：

- 本项目 `README.md` 的“关键决策”或“当前状态”
- 必要时追加到上级 `../全局复利与踩坑日志.md`

## 6. 默认执行原则

- Think Before Coding：先判断目标、边界、风险和成功标准。
- Simplicity First：优先使用能解决问题的最小方案。
- Surgical Changes：只改任务要求的内容。
- Goal-Driven Execution：完成后检查结果是否符合请求。
